# 멀티플레이 백엔드 기획 검토

> ai-trpg(싱글, Next.js+TS)를 **협동 1던전 멀티플레이**로 확장하기 위한 백엔드 웹서비스 기획.
> 목표: 안정적으로 대규모 트래픽을 처리하는 구조 (Java/Spring Boot 기반).

작성일: 2026-06-08

| 항목 | 결정 | 근거 |
|---|---|---|
| 1차 목적 | **실서비스 런칭** | 대규모 트래픽 = 실제 목표 |
| 멀티 형태 | **협동 1던전** | 여러 사람이 같은 던전에서 각자 파티원 조작, 실시간 동기화 |

## 1. 현황 (싱글플레이 구조)

- **서버 = 무상태**. `GameState`를 클라이언트가 보유, 매 행동마다 POST 왕복. 서버는 순수함수: `gameState + action → 엔진 → LLM → 새 gameState`.
- **DB 없음**. 영속화 전무. 유일한 서버 상태는 인메모리 `apiKeySessions` Map.
- **엔진 = 결정적 TS 로직** (`src/lib/engine`: dice/combat/movement). **LLM = 나레이션 전담**. 책임분리 명확 (설계문서 §11.1, §11.6).
- 행동 1회 = LLM 1콜(5~30초, 유료). BYOK라 비용 유저 부담.
- 멀티플레이는 설계문서 §부록 B에 "향후 확장 후보"로만 기재.

## 2. 목표 아키텍처

```
[클라(Next.js)] ──WebSocket──┐
                             ▼
                   [Nginx/ALB · 스티키 라우팅]
                             ▼
         ┌────────── Spring Boot (모놀리식) ──────────┐
         │  HTTP: 인증/로비/매치메이킹 (무상태)        │
         │  WS:   룸 채널 (룸별 단일 라이터)           │
         │  Engine(Java 포팅): dice/combat/movement   │  ← 결정적, 서버 권위
         │  LLM Orchestrator: BYOK 호출 + 스트리밍     │
         └────┬───────────────┬───────────────┬───────┘
              ▼               ▼               ▼
          [Redis]        [Postgres]      [LLM API]
       라이브 룸상태      유저/세이브       (호스트 키)
       Pub/Sub 백플레인   룸 메타
       락/레이트리밋
```

## 3. 핵심 설계 판단

### ① 서버 권위 상태 (최대 변경점)
현재는 클라가 `GameState` 보유·왕복 → 멀티에서 **불가** (치팅·동기화 충돌). 협동 1던전은 하나의 공유 상태를 서버가 소유해야 함.

- **Redis = 라이브 권위 상태** (저지연 read/write).
- **Postgres = 영속 스냅샷** (`docs/save-load-plan.md` 이미 존재 → 연계).
- 클라는 *의도(intent)* 만 전송. 상태는 절대 보유하지 않음.

### ② 룸 = 단일 라이터 (동시성 핵심)
협동이라 2명 이상이 동시 행동 가능 → 공유 상태 레이스 컨디션.

- 룸별로 모든 intent를 한 핸들러로 직렬화 (액터 모델 또는 Redis 분산락).
- 턴 진행 중 룸 잠금.
- 스케일: **룸을 인스턴스에 샤딩** (roomId 일관해싱·스티키). 무상태 HTTP(로비)는 단순 수평 확장.

### ③ 엔진 Java 포팅 = 테스트가 스펙
`src/lib/engine/*.ts` + **모든 `*.test.ts`를 JUnit으로 함께 포팅**.

- 테스트가 게임 규칙 명세 (combat/dice/exploration/inspiration).
- 패리티 보장 안 하면 밸런스 깨짐.
- 결정적 RNG는 서버 측·재현가능해야 함 (설계문서 §11.6). 시드 기반 RNG 모듈로 구현.
- 주의: TS↔Java 부동소수·정수 처리 차이.

### ④ LLM 핫패스 + BYOK = 가상스레드 + 스트리밍
행동마다 5~30초 유료 콜.

- 블로킹 SDK 그대로 쓰려면 **Java 21 가상 스레드** (WebFlux보다 단순, 기존 AI provider 구조 이식 쉬움).
- 나레이션 토큰을 룸 전원에게 WS 스트리밍.
- 룸당 in-flight LLM 1개 → 자연 백프레셔.

## 4. 남은 결정 (추천 동봉)

| 결정 | 추천 | 비고 |
|---|---|---|
| BYOK 협동 시 누구 키? | ⭐ **방장(host) 키로 세션 전체 나레이션** | 키는 서버 측에만 머무름(브로드캐스트 금지), 방장이 사용량 확인·동의. 대안: 페이어 라운드로빈(복잡) |
| 빈 슬롯 (1~2인 플레이) | ⭐ **AI가 남은 파티원 채움** (전사/에이미/실루엘라) | 중간 이탈 시 AI 인계 |
| 재접속/이탈 | ⭐ **프레즌스 + 재접속 유예 + AI takeover** | 협동 필수 기능 |

## 5. 단계적 빌드 (STUDY_PLAN 정합)

실서비스라도 STUDY_PLAN "초반 금지"(MSA·Sharding·Kafka)는 지킴. 모놀리식 시작, 트래픽 증거 나오면 분리.

| 단계 | 내용 | STUDY_PLAN 매핑 |
|---|---|---|
| **v0 협동 MVP** | Spring 모놀리식 + 엔진 포팅 + WS 룸 + Redis 상태 + Postgres 세이브 + 인증(JWT) | Phase 4·5·6·7·8·9 |
| **v1 스케일아웃** | 룸 샤딩 + Redis Pub/Sub 백플레인 + Nginx 스티키 + read replica | Phase 11 (1~5) |
| **v2 대규모** | 큐(영속 배치·분석) + 옵저버빌리티 + 서킷브레이커/백프레셔 | Phase 11 (4·7·8) |

## 6. 위험 / 주의

- **BYOK 키 보안**: 현재 인메모리 Map → 멀티인스턴스에선 불가. 암호화 저장·짧은 TTL·로그 금지·타 클라 전송 금지. 협동에선 호스트 키로 서버측 호출만 (키 노출 0).
- **BYOK 남용**: 방장 키가 동료들에 의해 소진 → 사용량 노출·동의 UI 필요.
- **LLM 지연이 턴 락**: 한 명 행동의 나레이션(30초) 동안 룸 대기 → UX 설계(스트리밍·타이핑 표시·동시 비전투 행동 허용 여부) 필요.
- **엔진 포팅 패리티**: TS↔Java 처리 차이. 테스트 포팅이 안전망.
